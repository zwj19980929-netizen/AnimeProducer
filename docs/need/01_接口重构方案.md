# AnimeProducer 接口重构方案

## 1. 文档目标

这份文档定义新的 API 设计方向，目标不是简单改路由，而是让接口天然支持新的交互体验：

1. 前端能基于接口直接搭建“阶段式工作台”。
2. 用户能看见清晰的下一步，而不是被一堆平级动作包围。
3. 所有长任务都能统一监控、取消、重试和追踪。
4. 资源边界清晰，减少页面到处拼接口的情况。

## 2. 当前接口存在的问题

## 2.1 资源边界不清

当前接口里有几类边界混用：

1. `character` 资源在 `/assets/characters`。
2. 角色扫描、合并、别名管理却放在 `/projects/{id}/chapters/*`。
3. 项目级 storyboard 和分集级 storyboard 各自一套动作，但没有统一抽象。

这会让前端难以回答一个简单问题：

“我现在在项目上下文中操作角色，应该请求哪个域？”

## 2.2 长任务模型不统一

当前长任务处理方式不一致：

1. 有些动作同步执行，前端只能拉长 timeout。
2. 有些动作返回 `Job`。
3. 有些任务完成后需要再查二次结果接口。

结果是前端交互体验不稳定：

1. 某些按钮点下去像“提交任务”。
2. 某些按钮点下去像“卡住等待”。
3. 某些任务能取消，某些不能。

## 2.3 页面缺少聚合接口

当前页面为了展示项目状态，需要分别查询：

1. project
2. chapters
3. episodes
4. characters
5. jobs
6. renders

这会导致：

1. 首屏请求多。
2. 页面逻辑分散。
3. 很难设计统一的 dashboard。

## 2.4 状态机无法支撑审核流

当前状态更偏“生产状态”，但缺少“审核状态”：

1. 分集规划草案和已确认规划没有明确区分。
2. 角色扫描结果和已确认角色圣经没有明确区分。
3. 分镜生成完成和已审核可渲染没有明确区分。

## 3. 重构原则

## 3.1 资源和动作分离

约定如下：

1. `GET/POST/PATCH/DELETE` 资源接口只负责数据。
2. `POST .../actions/*` 统一负责触发动作。
3. 所有 `actions` 返回统一的 `OperationResponse`。

## 3.2 长任务全部异步

以下动作全部异步化：

1. 导入并解析书籍
2. 批量章节分析
3. 角色扫描
4. 角色候选图生成
5. 批量角色变体生成
6. 分集规划
7. 分镜生成
8. 渲染
9. 批量上传 OSS

## 3.3 项目级与分集级职责明确

建议统一原则：

1. 项目级只负责全局资产、全局摘要和总览。
2. 分集级负责 storyboard、render、delivery。
3. 项目级“一键全流程”保留为高级批处理动作，不再作为默认主路线。

## 3.4 所有页面优先消费聚合接口

新增聚合接口：

1. `GET /projects/{project_id}/dashboard`
2. `GET /projects/{project_id}/workspace`
3. `GET /projects/{project_id}/production-board`

前端页面优先消费聚合结果，局部编辑再落到明细资源接口。

## 4. 新的核心资源模型

## 4.1 Project

项目是总容器，不再承载过多流程状态，而是承载全局信息。

建议字段：

```json
{
  "id": "proj_xxx",
  "name": "凡人修仙传 S1",
  "description": "第一季改编项目",
  "style_preset": "国风热血动画",
  "status": "ACTIVE",
  "current_stage": "EPISODE_PLANNING",
  "completion_rate": 0.42,
  "created_at": "2026-03-27T10:00:00Z",
  "updated_at": "2026-03-27T10:00:00Z"
}
```

建议新增：

1. `current_stage`
2. `completion_rate`
3. `last_operation_id`
4. `last_blocker`

## 4.2 ProjectStageSummary

用于仪表盘展示。

```json
{
  "key": "CHARACTER_BIBLE",
  "label": "角色圣经",
  "status": "IN_PROGRESS",
  "progress": 0.6,
  "metrics": {
    "characters_total": 12,
    "characters_confirmed": 8
  },
  "blockers": [
    "4 个角色尚未设置锚点图"
  ],
  "primary_action": {
    "label": "继续确认角色",
    "target": "/projects/proj_xxx/characters"
  }
}
```

状态建议统一为：

1. `NOT_STARTED`
2. `IN_PROGRESS`
3. `READY_FOR_REVIEW`
4. `BLOCKED`
5. `COMPLETED`
6. `FAILED`

## 4.3 SourceBook

把“书籍信息”和“上传动作”分开。

字段：

```json
{
  "id": "book_xxx",
  "project_id": "proj_xxx",
  "title": "凡人修仙传",
  "author": "忘语",
  "genre": "玄幻",
  "source_type": "TXT",
  "upload_status": "COMPLETE",
  "total_chapters": 120,
  "uploaded_chapters": 120,
  "total_words": 3200000,
  "summary": "...",
  "suggested_episode_count": 24
}
```

## 4.4 Chapter

章节是内容资源，不应承载过多动作语义。

新增建议：

1. `analysis_status`
2. `review_status`
3. `has_character_candidates`
4. `has_episode_assignment`

## 4.5 Character

建议统一挂在项目下：

`/projects/{project_id}/characters`

字段建议增加：

1. `asset_status`: `EMPTY / CANDIDATES_READY / ANCHOR_READY / VARIANTS_READY / TRAINING_READY`
2. `review_status`: `PENDING / APPROVED / REJECTED`
3. `source_chapters`: 角色来源章节
4. `issues`: 别名冲突、身份疑似冲突等

## 4.6 EpisodePlanDraft

这是当前系统缺失的关键对象。

分集规划不应直接等于 `Episode`，因为 AI 规划和最终确认是两回事。

```json
{
  "id": "plan_xxx",
  "project_id": "proj_xxx",
  "status": "DRAFT",
  "version": 3,
  "suggestions": [
    {
      "episode_number": 1,
      "title": "初入仙途",
      "start_chapter": 1,
      "end_chapter": 5,
      "estimated_duration_minutes": 23.5
    }
  ],
  "reasoning": "..."
}
```

状态建议：

1. `DRAFT`
2. `CONFIRMED`
3. `ARCHIVED`

## 4.7 Episode

`Episode` 是正式生产单元。

建议字段：

1. `planning_source_id`
2. `storyboard_status`
3. `render_status`
4. `review_status`
5. `delivery_status`
6. `latest_storyboard_version`
7. `latest_delivery_id`

## 4.8 Storyboard

建议补充明确对象，不再只靠 `Shot` 列表隐式存在。

```json
{
  "id": "sb_xxx",
  "episode_id": "ep_xxx",
  "version": 2,
  "status": "READY_FOR_REVIEW",
  "shot_count": 38,
  "estimated_duration_minutes": 22.8,
  "created_by": "AI",
  "created_at": "..."
}
```

## 4.9 Shot

保留，但建议补充：

1. `storyboard_id`
2. `shot_code`: 如 `E01-S012`
3. `review_status`
4. `render_status`
5. `scene_id`
6. `notes`
7. `source_chapter_numbers`

## 4.10 Operation

统一所有异步动作。

```json
{
  "id": "op_xxx",
  "project_id": "proj_xxx",
  "type": "CHARACTER_SCAN",
  "status": "RUNNING",
  "progress": 0.45,
  "target_type": "PROJECT",
  "target_id": "proj_xxx",
  "created_at": "...",
  "started_at": "...",
  "finished_at": null,
  "message": "正在扫描第 4/10 章",
  "result": {},
  "error": null
}
```

状态建议：

1. `QUEUED`
2. `RUNNING`
3. `SUCCEEDED`
4. `FAILED`
5. `CANCELLED`

## 5. 新的接口分组

## 5.1 项目与聚合

### `GET /projects`

项目列表，支持筛选和分页。

### `POST /projects`

创建项目。

### `GET /projects/{project_id}`

项目基础信息。

### `PATCH /projects/{project_id}`

编辑项目信息。

### `GET /projects/{project_id}/dashboard`

项目仪表盘聚合接口。

返回：

1. 项目基础信息
2. 各阶段状态
3. 关键计数
4. 最近操作
5. 最近失败
6. 下一步建议

### `GET /projects/{project_id}/workspace`

项目工作台入口聚合接口，给前端首屏直接使用。

建议包含：

1. `dashboard`
2. `book_summary`
3. `chapter_summary`
4. `character_summary`
5. `episode_summary`
6. `render_summary`

## 5.2 书源与章节

### `GET /projects/{project_id}/book`

获取书籍元数据。

### `PATCH /projects/{project_id}/book`

编辑书籍元数据。

### `POST /projects/{project_id}/book/actions/upload-and-parse`

上传并解析整本书，返回 `OperationResponse`。

### `POST /projects/{project_id}/book/actions/analyze`

分析全书摘要和总集数建议。

### `GET /projects/{project_id}/chapters`

章节列表，支持：

1. `page`
2. `page_size`
3. `analysis_status`
4. `review_status`
5. `episode_number`

### `POST /projects/{project_id}/chapters`

新增单章或批量新增。

### `PATCH /projects/{project_id}/chapters/{chapter_number}`

编辑章节。

### `POST /projects/{project_id}/chapters/actions/analyze-batch`

批量章节分析，返回 operation。

### `POST /projects/{project_id}/chapters/actions/scan-characters`

从指定章节范围扫描角色，返回 operation。

## 5.3 角色圣经

### `GET /projects/{project_id}/characters`

角色列表。

支持筛选：

1. `asset_status`
2. `review_status`
3. `has_anchor`
4. `keyword`

### `POST /projects/{project_id}/characters`

手工新增角色。

### `GET /projects/{project_id}/characters/{character_id}`

角色详情。

### `PATCH /projects/{project_id}/characters/{character_id}`

编辑角色基本信息。

### `POST /projects/{project_id}/characters/{character_id}/actions/generate-candidates`

异步生成候选图。

### `POST /projects/{project_id}/characters/{character_id}/actions/generate-variants`

异步生成变体图。

### `POST /projects/{project_id}/characters/{character_id}/actions/set-anchor`

设置锚点图。

### `POST /projects/{project_id}/characters/{character_id}/actions/approve`

将角色标记为“可用于生产”。

### `POST /projects/{project_id}/characters/actions/merge`

角色合并。

### `POST /projects/{project_id}/characters/actions/add-alias`

新增别名。

## 5.4 分集规划

### `POST /projects/{project_id}/episode-plans/actions/generate`

生成新的分集草案，返回 operation。

### `GET /projects/{project_id}/episode-plans`

历史规划草案列表。

### `GET /projects/{project_id}/episode-plans/{plan_id}`

某次规划详情。

### `POST /projects/{project_id}/episode-plans/{plan_id}/actions/confirm`

把规划草案落地为正式 `episodes`。

### `POST /projects/{project_id}/episode-plans/{plan_id}/actions/archive`

归档旧草案。

## 5.5 分镜工作台

### `GET /projects/{project_id}/episodes`

分集列表。

### `GET /projects/{project_id}/episodes/{episode_number}`

分集详情。

### `PATCH /projects/{project_id}/episodes/{episode_number}`

编辑分集信息。

### `POST /projects/{project_id}/episodes/{episode_number}/storyboards/actions/generate`

为单集生成 storyboard，返回 operation。

### `GET /projects/{project_id}/episodes/{episode_number}/storyboards`

分镜版本列表。

### `GET /projects/{project_id}/episodes/{episode_number}/storyboards/{storyboard_id}`

分镜详情。

### `GET /projects/{project_id}/episodes/{episode_number}/shots`

当前激活版本的镜头列表。

### `PATCH /projects/{project_id}/episodes/{episode_number}/shots/{shot_id}`

人工编辑镜头。

### `POST /projects/{project_id}/episodes/{episode_number}/storyboards/{storyboard_id}/actions/approve`

审核通过，允许进入渲染。

## 5.6 渲染中心与交付

### `POST /projects/{project_id}/episodes/{episode_number}/renders/actions/start`

启动单集渲染，返回 operation。

### `GET /projects/{project_id}/episodes/{episode_number}/renders`

该集渲染批次列表。

### `GET /projects/{project_id}/episodes/{episode_number}/renders/{render_batch_id}`

渲染批次详情，包括 shot renders。

### `GET /projects/{project_id}/deliveries`

项目交付物列表。

### `GET /projects/{project_id}/deliveries/{delivery_id}`

单个交付详情。

## 5.7 操作中心

### `GET /projects/{project_id}/operations`

项目内所有操作列表。

### `GET /operations/{operation_id}`

操作详情。

### `POST /operations/{operation_id}/actions/cancel`

取消操作。

### `POST /operations/{operation_id}/actions/retry`

重试操作。

### `GET /ws/projects/{project_id}`

项目级实时事件流。

### `GET /ws/operations/{operation_id}`

操作级实时事件流。

## 7. 与现有接口的映射建议

## 7.1 保留但弱化的旧接口

以下旧接口可以保留一段时间用于兼容：

1. `/projects/{id}/assets/build`
2. `/projects/{id}/assets/build-from-chapters`
3. `/projects/{id}/storyboard/generate`
4. `/projects/{id}/pipeline/start`

但在新前端中不再作为主入口。

## 7.2 建议迁移规则

### 旧接口

`POST /projects/{id}/chapters/extract-characters`

### 新接口

`POST /projects/{id}/chapters/actions/scan-characters`

理由：

1. 表达“对章节执行动作”。
2. 与其他动作命名统一。
3. 不把角色域逻辑硬塞到 chapter 子路由下。

### 旧接口

`POST /assets/characters/{character_id}/generate-reference`

### 新接口

`POST /projects/{project_id}/characters/{character_id}/actions/generate-candidates`

理由：

1. 角色必须明确属于项目上下文。
2. `reference` 对用户来说是技术词，`candidates` 更贴近工作流。

### 旧接口

`POST /projects/{project_id}/episodes/plan`

### 新接口

`POST /projects/{project_id}/episode-plans/actions/generate`

理由：

1. 把“草案”与“正式分集”分开。
2. 后续可支持多版本规划历史。

## 8. 实施优先级

## P0

1. 新增 `dashboard` 聚合接口。
2. 抽象统一 `operation` 响应。
3. 把书籍上传、章节分析、分集规划、分镜生成、渲染统一改为异步动作。

## P1

1. 角色接口迁移到项目上下文。
2. 新增 `episode_plan_draft` 概念。
3. 增加 storyboard 版本对象。

## P2

1. 增加 review status。
2. 增加 delivery/version。
3. 增加高级批处理接口。

## 9. 前端收益

重构后前端可以明显简化：

1. 项目首页只调一次 `dashboard`。
2. 每个页面只处理一个领域。
3. 长任务统一显示成操作卡片。
4. 页面不再依赖大量按钮显示条件拼装。
5. UI 可以自然变成“阶段推进式工作台”。
