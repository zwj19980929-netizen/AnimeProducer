# AnimeProducer 交互与产品重构总览

## 1. 结论先行

你的核心架构思路是对的，尤其是这三点值得保留：

1. 以角色资产一致性为前置条件，而不是先盲目出镜头。
2. 用章节/分镜把长文本转成结构化生产对象。
3. 把渲染、合成、任务状态拆成后台能力，而不是全部压在前端。

当前问题不在“能力不够”，而在“产品入口和信息组织不友好”：

1. 用户进入项目后，看见的是一组并列 Tab，而不是一条清晰的制作路径。
2. 同时存在项目级流程和分集级流程，用户不知道应该走哪条主线。
3. 接口层把 CRUD、长任务、扫描动作、资源管理混在一起，前端只能堆按钮。
4. UI 视觉是通用暗色 SaaS 风格，没有体现“动画制作台”的专业感和流程感。

## 2. 本轮重构原则

本轮不推翻你的后端能力，只做三件事：

1. 明确产品主线。
2. 重构接口语义。
3. 重新设计信息架构和 UI。

建议把产品主线固定为：

`项目 -> 书源 -> 章节 -> 角色圣经 -> 分集规划 -> 分镜 -> 渲染 -> 成片交付`

其中：

1. `Project` 是协作容器。
2. `Episode` 是制作单元。
3. `Shot` 是执行单元。
4. `Character` 是跨章节、跨分集复用的核心资产。

## 3. 当前代码暴露出的主要问题

### 3.1 信息架构问题

当前项目详情页把 `概览 / 书籍 / 章节 / 分集 / 角色 / 分镜 / 渲染 / 输出` 全部平铺为同级入口。  
这对开发方便，但对用户有三个副作用：

1. 用户不知道第一步该点哪里。
2. 用户看不到当前卡点在哪里。
3. 用户无法理解哪些页面是“准备数据”，哪些页面是“正式生产”。

### 3.2 流程主线冲突

当前代码同时支持两条路径：

1. 项目级：直接构建资产 -> 生成全项目 storyboard -> 启动全项目 pipeline。
2. 分集级：上传章节 -> 规划分集 -> 对单集生成 storyboard -> 渲染单集。

从产品角度看，主线应该只有一条。  
建议保留分集级作为默认主流程，项目级“一键全流程”下沉为高级模式或批处理模式。

### 3.3 状态表达不够

现在 `ProjectStatus` 更像“粗颗粒进度标签”，无法回答下面这些关键问题：

1. 章节是否已完成解析但未完成分析？
2. 角色是否已扫描但仍有待确认别名/锚点图？
3. 分集规划是否存在草案但还未确认？
4. 某一集是否已出分镜但未审核？

这会直接导致前端按钮启用条件很难设计，也难以给用户“下一步建议”。

### 3.4 接口职责混乱

典型问题包括：

1. 角色 CRUD 在 `/assets/characters`，但角色扫描、合并、别名又挂在 `/projects/{id}/chapters/*`。
2. 长耗时动作有的同步阻塞 HTTP，有的走异步 Job。
3. 项目级和分集级的 storyboard/render 接口命名不统一。
4. 前端缺少统一的 dashboard summary，只能自己拼多次请求。

### 3.5 UI 视觉问题

当前 UI 的问题不是“丑”，而是“不像生产工具”：

1. 风格偏通用后台，不像创作台。
2. 视觉层级不服务流程推进。
3. 关键对象之间缺少上下文关系。
4. 缺少“阶段感、审核感、产线感”。

## 4. 重构后的产品骨架

### 4.1 新的导航结构

建议把项目详情页改成左侧阶段式导航，而不是顶层多 Tab：

1. 仪表盘
2. 书源与章节
3. 角色圣经
4. 分集规划
5. 分镜工作台
6. 渲染中心
7. 交付输出

### 4.2 新的页面逻辑

每个页面只回答一个核心问题：

1. 仪表盘：我现在做到哪一步了，下一步是什么？
2. 书源与章节：原始内容是否已经被正确解析？
3. 角色圣经：人物是否已具备可复用资产？
4. 分集规划：章节如何切成可生产的集？
5. 分镜工作台：每一集的镜头结构是否可执行？
6. 渲染中心：生产任务是否顺畅，哪里失败了？
7. 交付输出：当前有哪些成片版本可预览和导出？

### 4.3 新的接口原则

接口层统一成两类：

1. 资源接口：负责查询和编辑数据。
2. 动作接口：负责触发长任务，统一返回 `operation/job`。

所有耗时操作都异步化，包括：

1. 上传并解析书籍
2. 批量章节分析
3. 角色扫描
4. 角色参考图生成
5. 分集规划
6. 分镜生成
7. 分集渲染

1. `01_接口重构方案.md`

建议阅读顺序也是这个顺序：

1. 先定对象模型和接口。
2. 再定产品需求与流程边界。
3. 最后根据接口和流程去做 UI 设计。

## 6. 落地建议

建议分三期做，而不是一次性重写：

### 第一期：接口与聚合层

1. 增加 `project dashboard summary` 接口。
2. 新增统一 `operations` 模型。
3. 明确 `project / chapter / character / episode / storyboard / delivery` 资源边界。

### 第二期：前端信息架构

1. 用左侧阶段导航替换大 Tab。
2. 先做仪表盘、书源与章节、角色圣经、分集规划四个页面。
3. 保留旧页面作为内部入口，逐步迁移。

### 第三期：体验增强

1. 增加审核流。
2. 增加失败修复入口。
3. 增加版本与批次概念。
